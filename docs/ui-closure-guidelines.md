# UI 收口执行规范

本规范吸收 `taste-skill` 的旧项目重做思路，但按本项目的 PC 端卡牌肉鸽定位收束：不做营销页，不追求移动端，不改变“暗黑童话纸牌剧场”的视觉方向。

## 核心原则

1. 每个界面先保证玩家能完成动作，再追求装饰。
2. 所有插画都必须放进固定尺寸的 `ArtFrame`，不能用绝对定位压在文字、血条、按钮之上。
3. 卡片、面板、弹窗必须有清晰的信息轨道：标题、资源、主要操作、次要说明不能混在同一个视觉层。
4. PC 端优先稳定：宽度、高度、grid 轨道、滚动区域都要明确，不能靠内容把布局撑开。
5. 美术资源只能加强识别，不能降低读数、目标选择和战斗意图的可读性。

## 页面级检查

- 顶栏：只显示全局状态和明确入口，不能出现多行拥挤。
- 战斗页：单位卡、手牌区、判定区、行动按钮、右侧栏都要在 1440px 宽度下完整可读。
- 宽屏布局：在 1600px 以上窗口中，主应用应主动展开到 1840px 上限，不能只占屏幕中央一小块；开始页应占用大部分首屏高度。
- 地图页：节点与连线必须是主视觉，资源入口不能抢占路线区域。
- 背包/卡包/制牌室：以表格化、可扫描为主，图片尺寸统一，不做大图展示。
- 商店/事件/结算：每个可点选项必须清楚显示成本、收益、不可选原因。
- 弹窗：只解释当前点击对象，不塞系统规则书全文。

## 组件级硬约束

- `ArtFrame` 内图片只能 `cover` 或 `contain`，外层必须 `overflow: hidden`。
- 单位卡片必须使用“内容区 + 图片区”的布局，禁止图片绝对定位到文字层。
- 按钮文字不能溢出；如果命令很长，缩短文案而不是缩小字体。
- 面板内不能再套大型浮动卡片；重复元素才使用卡片。
- 所有页面必须通过横向溢出检查。

## 验证规则

每次 UI 收口至少跑：

- `npm run typecheck`
- `npm run build`
- `npx playwright test tests/e2e/ui-layout.spec.js`

涉及战斗 UI 时，还要确认：

- 主角图、敌人图、侧栏头像不越出 `ArtFrame`。
- 单位图不与血条、数值、状态、意图、特性文本重叠。
- 多敌人时仍能点击选择主目标，并能打开意图/特性详情。

涉及布局适配时，还要确认：

- 1280、1366、1440 常见 PC 宽度无横向溢出。
- 1920、2048 宽屏下 `.app` 宽度至少使用 80% 可视宽度。
- 开始页在宽屏下不是小卡片，首屏高度利用率至少达到 70%。
